# 文档解析工作流

## 识别文档类型

| 文档类型 | 特征 | 解析重点 |
|----------|------|----------|
| PRD（产品需求文档） | 有背景、目标、功能列表章节 | 功能列表 + 验收标准章节 |
| 功能规格说明 | 详细描述单一功能的行为 | 每个行为描述即一条需求 |
| 用户故事集 | "作为...我希望...以便..." 格式 | 每个 Story 映射一条 FUNC 需求 |
| 设计稿说明 | 描述 UI 交互和视觉 | 提取 UI 需求为主 |
| 业务需求文档 | 描述业务流程和规则 | 转化为系统行为需求 |
| 非结构化文本 | 口语化描述、邮件、会议记录 | 逐句扫描，识别动词描述的行为 |

## 提取信号词

以下词语出现时，其后内容大概率是一条需求：

- **功能类**：支持、允许、提供、实现、处理、显示、计算、生成、发送
- **约束类**：必须、需要、应该、不允许、禁止、限制
- **用户行为类**：用户可以、玩家能够、当用户点击/滑动/输入

## 拆分粒度判断

**太粗（需要拆分）：**
> "实现完整的用户系统"
→ 拆为：注册、登录、找回密码、个人资料编辑……

**合适粒度：**
> "用户点击撤销按钮后，最后一步操作回退"
→ 直接作为一条 FUNC 需求

**太细（可以合并）：**
> "按钮颜色为蓝色" + "按钮字体为16px" + "按钮圆角为4px"
→ 合并为一条 UI 需求："确认按钮的视觉规范（蓝色、16px、圆角4px）"

## 模糊需求处理

遇到模糊描述时：
1. 保留原文，在验收标准中标注 `[待明确]`
2. 在需求池结尾添加"待澄清问题"章节列出这些条目
3. 不要自行补充或猜测具体数值/行为

示例：
```
- **描述**：系统加载速度要快
- **验收标准**：
  - [ ] [待明确] 加载时间上限未定义，需产品确认具体指标
```

## 各分类识别规则

**FUNC（功能需求）**：系统做某件事，有明确输入/输出/行为
**UI（UI需求）**：涉及布局、颜色、字体、动画、交互手势
**PERF（性能需求）**：有量化指标：时间、内存、帧率、并发数
**DATA（数据需求）**：数据格式、字段定义、存储方式、数据量
**NFR（非功能需求）**：其余约束，如平台兼容、安全、无障碍
