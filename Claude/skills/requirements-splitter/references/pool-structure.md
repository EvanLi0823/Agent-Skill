# 需求池结构规范

## 需求分类

| 类型代码 | 类型名称 | 说明 |
|----------|----------|------|
| FUNC | 功能需求 | 系统必须提供的具体功能行为 |
| UI | UI/UX需求 | 界面布局、交互方式、视觉规范 |
| PERF | 性能需求 | 响应时间、并发、内存等性能指标 |
| DATA | 数据需求 | 数据结构、存储、格式要求 |
| NFR | 非功能需求 | 安全、兼容性、可维护性等 |

## 优先级定义

| 级别 | 定义 | 示例 |
|------|------|------|
| P0 | 核心功能，缺失则产品不可用 | 用户登录、核心游戏逻辑 |
| P1 | 重要功能，影响主流程体验 | 撤销操作、音效开关 |
| P2 | 增强功能，改善体验但不阻断 | 统计展示、动画效果 |
| P3 | 可选功能，后期迭代项 | 主题皮肤、高级设置 |

## ID 命名规则

格式：`REQ-{类型代码}-{三位序号}`
示例：`REQ-FUNC-001`、`REQ-UI-003`、`REQ-PERF-001`

同一文档内，每个类型独立编号从 001 开始。

## 验收标准写法规范

- 必须可验证（能被测试或肉眼确认）
- 使用 "当...时，系统应..." 或 "用户可以..." 句式
- 不写模糊词：快速、友好、合适

**好的例子：**
- [ ] 点击撤销按钮后，最后一步操作回退，牌面恢复到上一状态
- [ ] 游戏结束时，胜利动画在 300ms 内开始播放

**差的例子：**
- [ ] 系统响应快速（没有量化）
- [ ] 界面美观（不可验证）
