# 禁令优先原则：创建技能和代理的规范

基于"14 条提示词秘籍"，本文件定义创建 Skill 和 Agent 时必须遵守的禁令优先原则。

## 核心思路

与其花时间告诉 AI "应该怎么做"，不如先列出它容易犯的错，写禁令。
用 NEVER/DO NOT/CRITICAL 替代软性指引，**每条禁令必须附原因（Why）**。

---

## Skill 文件结构模板

在 SKILL.md frontmatter 之后、正文之前插入：

```markdown
## 本技能禁令 (NEVER / DO NOT)

**NEVER [最高发的错误行为]**
原因：[违反会导致什么具体后果]

**DO NOT [次高发的错误行为]**
原因：[原因]

**CRITICAL [最关键的约束]**
原因：[原因]
```

---

## Agent 文件结构模板

在"角色定位"段落之后、"核心能力/职责"之前插入：

```markdown
## 角色禁令 (NEVER / DO NOT)

**NEVER [该角色最易犯的错]**
原因：[原因]

**DO NOT [次高发错误]**
原因：[原因]

## 沟通规范
- 不使用 emoji
- 先说结论，再说理由
- 不做防御性表述（如"这只是我的建议"）
- 不确定时说"需要进一步确认"，不编造结果
```

---

## 14 条原则对应的禁令模式

| 原则 | 在 Skill/Agent 中的写法 |
|------|------------------------|
| 1. 禁令代替指令 | 在文件顶部加 NEVER/DO NOT 列表，不只写"应该做什么" |
| 2. 设唱反调 | workflow 型 Skill 预写"懒惰借口反驳"表（见下方示例） |
| 3. 不画蛇添足 | 明确禁止 AI 添加未被要求的功能/注释/辅助方法 |
| 4. 如实汇报 | 禁止防御性表述，禁止夸大结果 |
| 5. 活可分但思考不行 | 委托子任务时必须附完整上下文，禁止只说"帮我做X" |
| 6. 不知道就说不知道 | 禁止猜测或编造 API/方法签名 |
| 7. 先看再改 | NEVER 在未读取文件的情况下修改代码 |
| 8. 一次授权非永久 | NEVER 把"用户没追问"理解为"持续授权" |
| 9. 每条禁令写清原因 | 每条 NEVER 必须有"原因：" 说明 |
| 10. 信息按需给 | 详细内容放 references/，主文件只放关键禁令 |
| 11. 沟通规范 | Agent 必须有沟通规范模块（无 emoji、先结论） |
| 12. 不同场景不同规则 | 为简单/正常/紧急场景设置不同规则集 |
| 13. 模块化 | 角色/禁令/规则/任务分清晰模块，不混写 |
| 14. 限制工具使用 | 明确哪些工具在哪些场景使用，禁止用通用命令替代专用工具 |

## Workflow 型 Skill：懒惰借口反驳表（模板）

```markdown
## 预设反驳：警惕以下懒惰借口

| 借口 | 反驳 |
|------|------|
| "这个任务很简单，跳过[某阶段]" | 简单任务的[某阶段]只需5分钟，跳过后的返工要数小时 |
| "用户说尽快，跳过质量检查" | 尽快不等于跳过质量，有问题的输出比慢慢做更耗时 |
| "编译通过了，说明没问题" | 编译只验证语法，不验证逻辑正确或架构符合设计 |
| "上次这样做是可以的" | 上次授权仅对上次有效，这次需要重新确认 |
```

## 禁令质量自检清单

创建或更新 Skill/Agent 后，必须检查：

- [ ] 每个 NEVER/DO NOT 都有"原因："说明
- [ ] 禁令针对该 Skill/Agent 的真实高发错误（不是泛泛而谈）
- [ ] P0 级别问题用 CRITICAL/NEVER，不用"建议"、"注意"等弱词
- [ ] Workflow 型 Skill 有"懒惰借口反驳"表
- [ ] Agent 有"沟通规范"模块（无 emoji、先结论）
- [ ] 禁令模块位于 frontmatter 之后、正文顶部
