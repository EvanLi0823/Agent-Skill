# ReAct 模式详解

> 来源：Prompting Guide AI、IBM
> 翻译日期：2026-04-13

---

## 什么是 ReAct？

**ReAct**（Reasoning + Acting，推理+行动）是由 Yao 等人在 2022 年提出的一个框架，它使大语言模型（LLM）能够"以交错的方式生成推理轨迹和任务特定的行动"。这种方法将内部推理与外部工具交互相结合，以获得更可靠的响应。

ReAct Agent 是一种采用"推理和行动"框架的 AI 系统，它将思维链（CoT）推理与外部工具使用相结合。正如 IBM 所解释的，这些 Agent "结合了思维链推理和外部工具使用"，以增强语言模型在代理工作流中处理复杂问题的能力。

---

## ReAct 的工作原理

ReAct 通过循环过程运作，包含三个核心阶段：

### 思考-行动-观察循环

1. **思考（Thought）阶段**
   - 模型生成基于思考的推理轨迹来分解问题和规划解决方案
   - 模型用自然语言表达逐步推理，将大目标分解为可管理的子任务

2. **行动（Action）阶段**
   - 基于思考，模型决定执行哪个外部工具或操作
   - Agent 执行预定义的操作——调用 API、使用外部工具或查询知识库

3. **观察（Observation）阶段**
   - 外部来源的结果为下一步推理提供信息
   - 每次行动后，模型重新评估进度，决定继续还是提供最终输出

这个循环不断重复，直到任务完成。该框架解决了思维链提示的一个关键限制："无法访问外部世界或无法更新其知识可能导致事实幻觉等问题。"

根据资源，Agent "根据新信息或先前步骤的结果动态调整其方法"，类似于人类通过迭代式问题解决来处理复杂任务。

循环终止要么通过最大迭代次数限制，要么通过最终答案的置信度阈值。

## ReAct 示例结构

典型的 ReAct 轨迹包括：

```
问题：[用户的问题]

思考1：我需要先找出...
行动1：搜索[查询词]
观察1：根据搜索结果...

思考2：现在我知道了...，接下来需要...
行动2：计算[表达式]
观察2：计算结果是...

思考3：基于以上信息，我可以得出...
最终答案：[综合结论]
```

## 何时使用 ReAct

ReAct 在以下场景中表现出色：

### 适用场景

1. **知识密集型任务**
   - 问答系统
   - 事实验证
   - 需要外部信息的任务

2. **决策场景**
   - 需要探索和规划的复杂环境
   - 多步骤问题解决

3. **需要动态适应的任务**
   - 初始推理必须根据检索到的信息进行调整
   - 不可预测的场景需要复杂推理

研究表明，"ReAct 与思维链结合通常优于所有其他提示方法"，使其成为需要外部验证的复杂推理任务的理想选择。

## ReAct 的主要优势

1. **通用性**
   - 无需预先配置模型即可使用各种外部工具和 API
   - 灵活集成不同类型的工具

2. **适应性**
   - 从过去的结果中学习以处理意外挑战
   - 动态调整策略

3. **可解释性**
   - 言语化的推理过程便于调试和优化
   - 可以追踪 Agent 的思考路径

4. **准确性**
   - 外部信息源显著减少幻觉
   - 相比纯推理方法更可靠

---

## ReAct vs. 函数调用（Function Calling）

### 函数调用
- ✅ 更快的执行速度
- ✅ 减少 token 使用
- ❌ 适合简单任务
- ❌ 灵活性较低

### ReAct
- ✅ 动态、不可预测场景下表现出色
- ✅ 需要复杂推理时更合适
- ✅ 逐步透明的思考过程
- ❌ 使用更多 token
- ❌ 执行速度相对较慢

**选择建议**：
- 简单、明确的任务 → 函数调用
- 复杂、需要推理的任务 → ReAct

## 快速开始

### 支持 ReAct 的框架

多个框架支持 ReAct 实现，开发者无需从头构建：

1. **LangGraph**
   - 提供预配置的 ReAct 模块
   - 强大的状态管理能力

2. **LlamaIndex**
   - 简单易用的 ReAct Agent
   - 适合文档问答场景

3. **BeeAI**
   - 专注于企业级应用
   - 完整的 ReAct 支持

### 实现步骤

1. **定义工具**
   ```python
   tools = [
       {
           "name": "search",
           "description": "搜索互联网获取信息",
           "parameters": {...}
       },
       {
           "name": "calculator",
           "description": "执行数学计算",
           "parameters": {...}
       }
   ]
   ```

2. **设计提示词模板**
   ```
   你有以下工具可用：
   {tool_descriptions}

   使用以下格式：
   思考：[你的推理]
   行动：[工具名称和参数]
   观察：[工具返回结果]
   ...（重复）
   最终答案：[你的答案]
   ```

3. **实现循环逻辑**
   - 解析 LLM 输出
   - 执行工具调用
   - 将结果反馈给 LLM
   - 重复直到获得最终答案

---

## 实践建议

1. **限制循环次数**
   - 设置最大迭代次数（通常 5-10 次）
   - 避免无限循环

2. **清晰的工具描述**
   - 工具名称要直观
   - 参数说明要详细
   - 提供使用示例

3. **错误处理**
   - 处理工具调用失败
   - 提供有用的错误信息
   - 允许 Agent 重试

4. **监控和日志**
   - 记录每个思考-行动-观察步骤
   - 分析 Agent 的决策过程
   - 优化工具选择逻辑
